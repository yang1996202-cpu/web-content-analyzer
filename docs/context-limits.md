# AI 上下文限制详解

## 什么是上下文窗口？

### 通俗理解

AI 的"记忆"不是无限的，而是一个**固定大小的笔记本**：

```
笔记本大小：约 128K-200K tokens（约 10-15 万汉字）

每轮对话都在这个笔记本里写：
- 系统提示（你是谁）
- 历史对话（之前说了什么）
- 当前问题（现在问什么）
- 工具返回（搜索/读取的结果）

写满了怎么办？擦掉最早的内容！
```

### 技术定义

```
上下文窗口（Context Window）=
模型能同时处理的 token 数量上限

Token ≈ 单词/汉字的片段
- 英文：1 词 ≈ 1.3 tokens
- 中文：1 字 ≈ 1.5 tokens
- 代码：1 行 ≈ 5-20 tokens
```

## 实际限制分析

### 第一层：单次工具返回限制

| 工具 | 返回限制 | 说明 |
|------|---------|------|
| WebSearch | ~2-8K tokens | 搜索结果摘要 |
| Read File | ~2K-4K tokens | 文件内容 |
| Run Command | ~1K-2K tokens | 命令输出 |

**影响：**
```
用户："读这个 10 万字文档"
AI 调用 Read File
    ↓
工具返回：只有前 4000 tokens（约 2500 汉字）
    ↓
AI 只能基于这 2500 字回答
    ↓
文档后面的内容完全不知道！
```

### 第二层：对话历史累积

```
第1轮：
用户 100 字 + AI 回答 200 字 = 300 字

第2轮：
[历史 300 字] + 用户 100 字 + AI 200 字 = 600 字

第3轮：
[历史 600 字] + 用户 100 字 + AI 200 字 = 900 字

...第 50 轮...
累积：15000 字
    ↓
接近上限！最早的几轮开始被"遗忘"
```

### 第三层：系统提示占用

```
系统提示（AI 的"人设"）：约 500-2000 tokens

实际可用：
总窗口 128K - 系统提示 2K = 126K

还要分给：
- 历史对话
- 当前问题
- 工具返回
```

## 量化标准（实测经验）

### 安全阈值

| 等级 | 汉字数 | Token 数 | 说明 |
|------|--------|---------|------|
| 🟢 安全 | < 3000 | < 4500 | 单次工具可完整返回 |
| 🟡 警告 | 3000-10000 | 4500-15000 | 可能截断尾部 |
| 🔴 危险 | > 10000 | > 15000 | 只能读前面 10-30% |
| 💥 炸裂 | > 50000 | > 75000 | 几乎只能读目录 |

### 换算公式

```javascript
// 汉字转 Token（估算）
function chineseToTokens(chineseChars) {
  return Math.ceil(chineseChars * 1.5);
}

// Token 转汉字（估算）
function tokensToChinese(tokens) {
  return Math.floor(tokens / 1.5);
}

// 示例
chineseToTokens(3000);  // ≈ 4500 tokens（安全）
chineseToTokens(10000); // ≈ 15000 tokens（警告）
```

## 实际案例分析

### GitHub README 页面

```
URL: https://github.com/openclaw/openclaw

实际大小：
- HTML 源码：~80KB
- 纯文本：~15000 汉字
- Token 估算：~22500 tokens

AI 实际读到：
- WebSearch 返回：~6000 汉字（前 40%）
- 被省略：~9000 汉字（后 60%）

遗漏内容：
- Windows 安装特殊说明
- 高级配置选项
- 故障排除指南
```

### 技术文档网站

```
URL: https://docs.example.com/api

实际大小：
- 单页：~50000 汉字
- 多页总计：~200000 汉字

AI 处理：
- 只能读当前页的前 20%
- 其他页面完全不读（除非给链接）

正确做法：
- 给具体章节链接
- 分多次询问
- 每次只问一个章节
```

## 如何估算网页大小？

### 方法 1：浏览器控制台（精确）

```javascript
// 在浏览器按 F12，输入：
const text = document.body.innerText;
console.log(`总字符: ${text.length}`);
console.log(`中文字: ${(text.match(/[\u4e00-\u9fa5]/g) || []).length}`);
console.log(`估算Token: ${Math.ceil(text.length * 1.5)}`);
```

### 方法 2：快速估算（粗略）

```
鼠标滚轮滚动：
- < 3 屏 → 大概 < 3000 字 → 🟢
- 3-10 屏 → 大概 3000-10000 字 → 🟡
- > 10 屏 → 大概 > 10000 字 → 🔴

（假设一屏约 800-1000 字）
```

### 方法 3：使用 Web Content Analyzer（推荐）

```bash
node analyze.js https://example.com

输出：
总字符数: 15,000 字
等级: 🔴 危险
建议: 必须拆分询问
```

## 优化策略

### 给 AI 信息的技巧

| 场景 | 技巧 | 示例 |
|------|------|------|
| 长文档 | 给具体章节链接 | `.../docs#installation` |
| 多页网站 | 给具体页面 | 不要给首页，给子页面 |
| 代码仓库 | 给具体文件 | `.../blob/main/README.md` |
| 论文 | 给 PDF 页码 | "第 3-5 页的方法部分" |

### 压缩信息的方法

```
原始：
"请阅读这篇 2 万字文档，告诉我所有配置选项"

优化：
"这篇文档里，关于'安装要求'的部分说了什么？
（文档链接：...，只看 Installation 章节）"

进一步压缩：
"文档说需要 Node 20+，还有其他要求吗？"
```

## 不同模型的差异

| 模型 | 上下文窗口 | 实际可用 |
|------|-----------|---------|
| GPT-3.5 | 16K tokens | ~10K |
| GPT-4 | 128K tokens | ~100K |
| Claude 3 | 200K tokens | ~150K |
| Kimi | 200K tokens | ~150K |

**注意：**
- 窗口大 ≠ 效果好
- 长上下文容易"走神"
- 关键信息放在**前面**更保险

## 总结

> **理解上下文限制 = 更好地使用 AI**

核心要点：
1. AI 记忆有限（约 10-15 万汉字）
2. 长内容会被截断（只读前面）
3. 工具返回也有限制（2-8K tokens）
4. 渐进式对话是最佳实践

判断标准：
- < 3000 字：直接给
- 3000-10000 字：指定章节
- > 10000 字：必须拆分

优化原则：
- 给具体链接（带锚点 #）
- 分多次询问
- 每次聚焦一个主题
